# CONCERNS
_Generated by gsd-codebase-mapper · focus: concerns_

**Analysis Date:** 2026-05-18

## Summary

The codebase is relatively clean with no TODO/FIXME comments and clear architectural intent, but carries several meaningful risks: the plant browser loads up to 2,000 records into client memory with all fields fetched, filtering happens entirely in-browser, and a drag-and-drop reorder UI exists with no implementation wired up. The anon Supabase key and production URL are hardcoded into a committed test file. Three orphaned empty migration files and a deferred `plant_native_counties` table add schema noise without app functionality.

---

## Performance — HIGH

### Full Plant Catalog Downloaded to Every Browser Session

**Problem:** `app/(app)/plants/page.tsx` fetches up to 2,000 plant rows with `select('*')` in two parallel range queries (lines 268–269). All filtering, searching, and pagination runs in-browser against this in-memory array. At ~1,100+ plants and growing, initial page load already transfers 100–200KB+ of JSON before first interaction.

**Files:** `app/(app)/plants/page.tsx:268-284`

**Impact:** As the catalog grows, load time and memory usage grow linearly. At 3,000+ plants the approach will be noticeably slow on mobile/low-end devices. The current 2×1000-row pagination cap will silently truncate the catalog at 2,000 plants with no warning to the user.

**Fix approach:** Move filtering to Supabase PostgREST queries (server-side WHERE/ilike). Keep a lightweight card projection (id, common_name, latin_name, sun, water, plant_type, height_min, height_max, image_url) — the full `select('*')` is not needed for the card grid.

---

### N+1 Query Pattern on "Add to List" Menu

**Problem:** Every time a user opens the add-to-list dropdown on a plant card, `PlantCard` fires a Supabase query to check which lists contain that plant (`plant_list_items` select filtered by `plant_id`). With a grid of 100 cards visible at once, opening menus in quick succession — or network flakiness — can cause overlapping redundant queries.

**Files:** `app/(app)/plants/page.tsx:101-110`, `app/(app)/plants/[id]/AddToListClient.tsx:29-38`

**Impact:** Each menu open costs a round-trip; there is no deduplication or caching. Minor at current scale but the pattern is duplicated across two components.

**Fix approach:** Fetch all plant-list memberships for the current user's lists in a single query at load time, store as a Map keyed by `plant_id`.

---

### `select('*')` on Presentation Pages

**Problem:** Both `app/presents/[shareId]/page.tsx` and `app/presents/[shareId]/reports/page.tsx` use `select('*, plant:plants(*)')` when fetching list items, pulling every plant column even though only a subset is rendered.

**Files:** `app/presents/[shareId]/page.tsx:31-34`, `app/presents/[shareId]/reports/page.tsx:18-23`

**Impact:** Unnecessary data transferred on every public presentation view. These pages are unauthenticated and intended for client-sharing — they benefit the most from lean queries.

**Fix approach:** Enumerate the specific columns needed per view.

## Security — HIGH

### Supabase Anon Key Hardcoded in Committed Test File

**Problem:** `tests/global-setup.ts` (line 11) contains the production Supabase anon key as a string literal and the production Supabase project URL (line 10) hardcoded. This file is committed to the repository.

**Files:** `tests/global-setup.ts:9-11`

**Impact:** The anon key is public by Supabase design (it enforces RLS, not authentication), but committing it alongside the project URL in source reduces the barrier for abuse. More critically, it sets a precedent — if a service role key were added here by mistake, it would be immediately compromised.

**Fix approach:** Move both values to `.env.test.local` (which is already gitignored and already used in the same file for test credentials) and read them via `process.env`.

---

### `plant_list_items` SELECT Policy Exposes All Users' List Contents

**Problem:** The RLS policy `"List items are publicly readable"` on `plant_list_items` uses `USING (true)`, meaning any authenticated or anonymous caller can query any user's list items if they know the `list_id` UUID. The `plant_lists` SELECT policy is also `USING (true)`.

**Files:** `supabase/migrations/20260515024609_create_plant_lists_table.sql:42-44`

**Impact:** This is intentional for the public presentation feature (anyone with a `share_id` can view the presentation), but the current policy does not distinguish between public share access and direct list UUID access. A caller who guesses or obtains a `list_id` UUID can enumerate all plants in that list without going through the `share_id` flow. UUIDs are hard to guess but not secret.

**Fix approach:** If the intent is public sharing only through `share_id`, the `plant_list_items` policy should restrict direct `list_id` access to `owner_id = auth.uid()` OR join-check through `plant_lists.share_id` being known. Alternatively, document the open-read intent explicitly.

## Reliability — MEDIUM

### Drag-and-Drop UI with No Implementation

**Problem:** The list editor (`app/(app)/lists/[id]/page.tsx:79`) renders a drag handle icon (`⠿`) with `cursor-grab` styling on every plant item, strongly implying reorder functionality. There is no drag event handler, no DnD library, and no `sort_order` update mutation anywhere in the app code.

**Files:** `app/(app)/lists/[id]/page.tsx:79`

**Impact:** Users will attempt to drag and nothing will happen. The `sort_order` column exists in the schema and is used for ordering on display, but can only be set at insert time — there is no way to reorder items once added.

**Fix approach:** Either implement drag-and-drop (e.g., `@dnd-kit/sortable`) with an update mutation, or remove the cursor-grab handle until the feature is built.

---

### `NEXT_PUBLIC_SITE_URL` Not Set Causes Wrong Share URL in Production Copy

**Problem:** `app/(app)/lists/[id]/page.tsx:30` falls back to `http://localhost:3000` when `NEXT_PUBLIC_SITE_URL` is undefined. This env var is not listed in `CLAUDE.md`'s required env vars section and is not in the documented stack. If not set on Vercel, the "Copy share link" button will copy a `localhost` URL.

**Files:** `app/(app)/lists/[id]/page.tsx:30`

**Impact:** The `CopyShareUrl` component sends users a broken localhost link when `NEXT_PUBLIC_SITE_URL` is missing. Since Vercel exposes `VERCEL_URL` automatically, this could use `https://${process.env.VERCEL_URL}` as a fallback instead.

**Fix approach:** Add `NEXT_PUBLIC_SITE_URL` to the documented required env vars in `CLAUDE.md`, set it in the Vercel dashboard to `https://permacultureplantpicker.com`, or use `VERCEL_URL` as fallback.

---

### No Error Handling on Plant Load Failure

**Problem:** `app/(app)/plants/page.tsx` `load()` function (lines 265–283) has no try/catch and no error state. If either Supabase range query fails (network error, 5xx), `plants` stays `[]` and the UI shows "No plants match your filters" with no indication of failure.

**Files:** `app/(app)/plants/page.tsx:265-284`

**Impact:** Users see an empty state indistinguishable from a genuine "no results" situation, with no retry affordance.

**Fix approach:** Add an `error` state, display a user-visible error message with retry button on Supabase query failure.

---

### `handleAddToList` / `handleRemoveFromList` Mutations Are Fire-and-Forget

**Problem:** In `app/(app)/plants/page.tsx`, `handleAddToList` (line 328) and `handleRemoveFromList` (line 324) `await` Supabase mutations but discard errors entirely. If the insert/delete fails (RLS violation, network error), the UI optimistically shows the action as completed.

**Files:** `app/(app)/plants/page.tsx:324-344`, `app/(app)/plants/[id]/AddToListClient.tsx:43-60`

**Impact:** Silent data inconsistency — the user thinks a plant was added/removed but it wasn't.

**Fix approach:** Surface errors via toast notification or inline error state; revert optimistic UI on failure.

### Massive Single-File Page Component

**Problem:** `app/(app)/plants/page.tsx` is 654 lines, containing `FilterSection`, `PlantCard`, `PlantsPageInner`, and all filter/list mutation logic in one file. The `PlantsPageInner` component manages 12+ `useState` values and 4 `useEffect` hooks.

**Files:** `app/(app)/plants/page.tsx`

**Impact:** Difficult to test individual concerns. Adding new filter dimensions or card actions means modifying an already-complex component. The filter state management has no debouncing — every keystroke in the search box triggers `applyFilters` via `useCallback` + `useEffect`.

**Fix approach:** Extract `PlantCard` and `FilterSection` to their own files. Extract filter state into a custom hook `usePlantFilters`. Add debounce (e.g., 200ms) on the search input.

### Filter UI Logic Duplicated Between Desktop Sidebar and Mobile Drawer

## Test Coverage — LOW

### Test Suite Requires Production Data Shape

**Problem:** Several `plants.spec.ts` tests (`search for "lavender"`, `filter by "full sun"`) assert specific counts or expect a plant named "Lavender" to exist with `count === 1`. These tests will fail if production data changes (plant added with "lavender" in the name, lavender deleted, etc.).

**Files:** `tests/plants.spec.ts:29-35`

**Impact:** Fragile tests that can break due to data changes unrelated to code changes.

**Fix approach:** Test that search narrows results rather than asserting exact counts or specific plant names. Use `expect(count).toBeGreaterThan(0)` patterns consistent with other tests in the file.
