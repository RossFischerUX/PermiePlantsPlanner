# Architecture
_Generated by gsd-codebase-mapper · focus: arch_

## Summary

Next.js 14 App Router monolith backed by Supabase (PostgreSQL + Auth + RLS). The app uses a clear server/client split: RSCs fetch data directly from Supabase and render HTML; small `'use client'` islands handle interactive mutations. There are no API routes for data — all reads and writes go through the Supabase JS client in either server or browser context.

## Route Groups and Their Roles

**`app/(app)/` — authenticated app shell**
- `app/(app)/layout.tsx` — sticky nav (fetches `auth.getUser()` server-side), footer. The route group name is invisible to the URL router.
- `app/(app)/page.tsx` — marketing/home page (RSC, fetches hero plants from Supabase)
- `app/(app)/plants/page.tsx` — plant browser (`'use client'`, loads all plants client-side, filters in memory)
- `app/(app)/plants/[id]/page.tsx` — plant detail (RSC, single Supabase fetch)
- `app/(app)/lists/page.tsx` — My Lists dashboard (RSC, redirects unauthenticated users)
- `app/(app)/lists/[id]/page.tsx` — list editor (RSC, guards by `owner_id`)
- `app/(app)/auth/` — login, signup, forgot-password, reset-password, signout pages (all `'use client'`)

**`app/presents/` — public shareable views, no nav/footer**
- `app/presents/[shareId]/page.tsx` — plant grid presentation (RSC, fetches by `share_id`)
- `app/presents/[shareId]/reports/page.tsx` — water/bloom/season report tables (RSC)

**`app/api/auth/callback/route.ts` — auth callback**
- Handles both `token_hash` + `verifyOtp` (email links) and PKCE `code` + `exchangeCodeForSession` flows.
- Token-hash path is preferred; PKCE is a fallback.

## Data Flow

### Public plant browse (RSC path)
1. Browser requests `/plants/[id]`
2. Next.js renders `app/(app)/plants/[id]/page.tsx` on the server
3. Page calls `lib/supabase/server.ts` `createClient()` → reads cookie store → creates `@supabase/ssr` server client
4. Single `supabase.from('plants').select('*').eq('id', id)` query runs server-side
5. HTML returned to browser; no JS required for the read

### Plant browser (client-side filter path)
1. Browser loads `app/(app)/plants/page.tsx` (`'use client'`)
2. `useEffect` on mount fires two parallel Supabase queries: rows 0–999 and 1000–1999 of `plants` table, plus `auth.getUser()` to conditionally fetch the user's lists
3. All filtering (sun, water, type, zones, native state, bloom months, etc.) runs in-memory via `useCallback`/`useEffect`
4. Filter state is synced to URL query params via `router.replace` so browser back button restores the filter set
5. Plant-to-list mutations (`INSERT`/`DELETE` on `plant_list_items`) go directly to Supabase from the browser

### List mutation (client component path)
1. `NewListForm.tsx`, `ListItemActions.tsx`, `AddToListClient.tsx` are `'use client'`
2. They call `lib/supabase/client.ts` `createClient()` which returns a `createBrowserClient` instance
3. After a write, they call `router.refresh()` (Next.js App Router cache invalidation) to re-run the parent RSC and re-fetch updated data

### Auth flow
1. Signup/login pages (`'use client'`) call `supabase.auth.signUp` / `supabase.auth.signInWithPassword`
2. Email magic links send `token_hash` — user clicks → `app/api/auth/callback/route.ts` calls `supabase.auth.verifyOtp({ token_hash, type })`
3. On success, redirect to `/` (or `?next=` param)
4. `app/(app)/layout.tsx` (RSC) calls `supabase.auth.getUser()` on every navigation to determine nav state

### Public presentation flow
1. Landscape professional shares `/presents/[shareId]`
2. RSC fetches `plant_lists` by `share_id` (no auth required — RLS policy allows public SELECT)
3. Fetches `plant_list_items` joined with `plants` ordered by `sort_order`
4. Renders a no-nav, no-footer standalone page

## State Management

There is no global state management library. State is handled by:

- **URL query params** — plant browser filter state (`/plants?sun=full+sun&zones=9a,9b`), synced via `router.replace`
- **React `useState`** — local component state (filter toggles, modal visibility, form inputs, loading flags)
- **Supabase session cookies** — auth state, managed by `@supabase/ssr`; read server-side in layouts/RSCs, set via the auth callback route
- **`router.refresh()`** — triggers RSC re-render after client mutations to re-sync server-fetched data

## Database Schema

**`plants`** — master catalog, publicly readable
- `id UUID`, `common_name TEXT`, `latin_name TEXT`, `description TEXT`
- `sun TEXT` — enum: `full sun | part shade | full shade`
- `water TEXT` — enum: `low | moderate | high`
- `plant_type TEXT` — enum: `shrub | tree | perennial | groundcover | vine | grass`
- `bloom_months TEXT[]`, `season_of_interest TEXT[]`, `permaculture_uses TEXT[]`
- `native_states TEXT[]` — e.g. `['CA', 'OR']`
- `usda_zone_min INTEGER`, `usda_zone_max INTEGER` — half-zone encoding (9a=18, 9b=19); see `lib/zones.ts`
- `usda_zones TEXT` — raw text, kept for backward compat
- `is_invasive BOOLEAN`, `notable_cultivars TEXT`
- RLS: SELECT policy `USING (true)` — fully public

**`plant_lists`** — user-owned lists
- `id UUID`, `owner_id UUID → auth.users(id)`, `title TEXT`, `description TEXT`
- `share_id TEXT UNIQUE DEFAULT substring(md5(random()::text), 1, 13)` — auto-generated 13-char share token
- RLS: SELECT public; INSERT/UPDATE/DELETE scoped to `auth.uid() = owner_id`

**`plant_list_items`** — join table
- `id UUID`, `list_id UUID → plant_lists(id)`, `plant_id UUID → plants(id)`
- `sort_order INTEGER`, `notes TEXT`
- RLS: SELECT public; write ops verify list ownership via sub-select

## Auth Architecture

Supabase email/password auth with custom email templates (`supabase/templates/`). Token-hash OTP flow is used for all email link operations (password reset, signup confirmation) rather than PKCE — this avoids cross-browser token consumption issues and Chrome Safe Browsing prefetch invalidation.

`NavUser.tsx` (`'use client'`) handles sign-out by calling `supabase.auth.signOut()` then `router.push('/auth/signout')`. The RSC layout re-reads auth state on each navigation, so nav items update without client-side auth subscriptions.

---

## Error Handling

No unified error boundary. Patterns per context:

- **RSCs:** `notFound()` from `next/navigation` for missing records; `redirect('/auth/login')` for unauthenticated access to protected pages
- **Client components:** local `useState` error string, displayed as `<p className="text-red-600 text-sm">` inline below forms
- **Auth callback:** on failure, redirects to `/auth/login?error=auth`

## Anti-Patterns

### Client-side full-table load in plant browser
**What happens:** `app/(app)/plants/page.tsx` fetches up to 2000 plant rows in two sequential Supabase calls and filters entirely in-memory.
**Why it's wrong:** All filtering logic runs in the browser; scales poorly beyond ~2000 plants and sends unnecessary data to the client.
**Do this instead:** Push filters to Supabase query params (`.eq()`, `.contains()`, `.overlaps()`, `.gte()/.lte()` for zones) and paginate server-side.

### Duplicate display constants across files
**What happens:** `SUN_ICONS`, `WATER_ICONS`, `SUN_LABELS`, `WATER_LABELS` are redefined in `app/(app)/plants/page.tsx`, `app/(app)/plants/[id]/page.tsx`, `app/(app)/page.tsx`, and `app/presents/[shareId]/page.tsx`.
**Why it's wrong:** Changes to labels must be made in multiple places.
**Do this instead:** Extract to `lib/plant-labels.ts` and import from there.
