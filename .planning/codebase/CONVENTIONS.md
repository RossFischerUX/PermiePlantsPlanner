# Coding Conventions
_Generated by gsd-codebase-mapper · focus: quality_

## Summary

TypeScript strict mode throughout with no `any` usage found in app or lib code. Styling is exclusively Tailwind utility classes using the Botanical Heritage design token set — no component library, no custom CSS files. The codebase distinguishes clearly between server and client components via file naming and the `'use client'` directive, with a strict rule on which Supabase client to use in each context.

---

## Naming Patterns

**Files:**
- Page files: always `page.tsx` (Next.js App Router convention)
- Layout files: always `layout.tsx`
- Client components extracted from pages: `PascalCase.tsx`, co-located in the same route directory (e.g., `NavUser.tsx`, `NewListForm.tsx`, `CopyShareUrl.tsx`, `ListItemActions.tsx`, `AddToListClient.tsx`)
- Lib utilities: lowercase with hyphens — `types.ts`, `zones.ts`, `us-states.ts`
- Supabase clients: `lib/supabase/server.ts`, `lib/supabase/client.ts`

**Components / functions:**
- Exported page/layout components: `PascalCase` named exports matching the file's role (e.g., `export default function ListsPage()`, `export default function AppLayout()`)
- Internal sub-components defined in the same file: `PascalCase` (e.g., `FilterSection`, `PlantCard`, `PlantsPageInner`, `InfoCell`)
- Event handlers: `handle` prefix — `handleSubmit`, `handleSignOut`, `handleRemove`, `handleAddToList`, `handleCreateList`
- Boolean state: descriptive noun — `loading`, `copied`, `drawerOpen`, `showMenu`, `creatingList`

**Variables / state:**
- `camelCase` throughout
- Filter state arrays named for the property they filter: `sun`, `water`, `types`, `months`, `zones`, `nativeState`
- Supabase data destructured immediately: `const { data: list }`, `const { data: { user } }`

**Types:**
- Domain types defined as `interface` in `lib/types.ts`: `Plant`, `PlantList`, `PlantListItem`
- Enum-like string unions defined as `type` aliases: `Sun`, `Water`, `PlantType`, `ForestGardenLayer`
- Lookup maps typed as `Record<string, string>`: `SUN_ICONS`, `WATER_ICONS`, `SUN_LABELS`, `WATER_LABELS`
- Constants: `SCREAMING_SNAKE_CASE` arrays for filter option lists (e.g., `SUN_OPTIONS`, `ZONE_LABELS`, `PERM_USE_OPTIONS`)

## TypeScript Usage

**Strict mode:** Enabled via `tsconfig.json` `"strict": true`. The `scripts/` directory is excluded from the TS build (`"exclude": ["node_modules", "scripts"]`).

**Type imports:** Use `import type` for type-only imports — `import type { Plant, PlantList } from '@/lib/types'`. Function/value imports use plain `import`.

**Non-null assertion `!`:** Used on environment variables and on values known to exist post-guard (e.g., `process.env.NEXT_PUBLIC_SUPABASE_URL!`). Not used speculatively.

**Optional chaining:** Preferred over null checks for nullable fields — `plant.latin_name?.toLowerCase()`, `p.bloom_months?.some(...)`.

**Type narrowing with `.filter(Boolean)`:** Used alongside an explicit type cast to strip nulls from arrays, e.g.:
```typescript
].filter(Boolean) as { label: string; value: string }[]
```

**`any` usage:** No explicit `: any` found in `app/` or `lib/`. One type cast workaround exists in `app/(app)/lists/page.tsx`: `(list.plant_list_items as { count: number }[])` to handle the Supabase aggregation response shape.

**Path alias:** `@/*` maps to the project root (configured in `tsconfig.json`). Use `@/lib/...`, `@/app/...` — never relative `../../` imports between top-level directories.

## Component Patterns

**Server vs. Client split:**

| Pattern | When | Example |
|---------|------|---------|
| Async server component (no directive) | Data fetching, auth guards, static rendering | `app/(app)/lists/page.tsx`, `app/(app)/plants/[id]/page.tsx` |
| `'use client'` component | Browser state, event handlers, `useRouter`, `useSearchParams` | `app/(app)/plants/page.tsx`, `NavUser.tsx`, `NewListForm.tsx` |

**Rule:** Use `lib/supabase/server.ts` (`createClient` async) in RSCs and route handlers. Use `lib/supabase/client.ts` (`createClient` sync) only in `'use client'` components.

**Auth guard pattern (server):**
```typescript
const { data: { user } } = await supabase.auth.getUser()
if (!user) redirect('/auth/login')
```

**Not-found pattern (server):**
```typescript
const { data: plant } = await supabase.from('plants').select('*').eq('id', id).single()
if (!plant) notFound()
```

**Suspense wrapper for client pages using `useSearchParams`:**
```typescript
export default function PlantsPage() {
  return (
    <Suspense fallback={<div className="...">Loading…</div>}>
      <PlantsPageInner />
    </Suspense>
  )
}
```
The exported default is a thin Suspense shell; the real implementation lives in `PlantsPageInner`.

**Client sub-components extracted from server pages:** When a server page needs isolated client interactivity, extract it to a sibling `PascalCase.tsx` file with `'use client'` at the top (e.g., `CopyShareUrl.tsx`, `ListItemActions.tsx`, `AddToListClient.tsx`).

**Data fetching in client components:** Inline async functions called inside `useEffect` — no SWR or React Query. Supabase client instantiated at component scope with `const supabase = createClient()`.

**Loading state:** Local `useState<boolean>` named `loading` (or `creatingList`). Buttons show ellipsis text while pending: `'Signing in…'`, `'Creating…'`, `'Creating account…'`.

**Error state:** Local `useState<string>('')` named `error`. Rendered as `<p className="text-red-600 text-sm">{error}</p>` directly in the form. Note: `text-red-600` is the one exception to the Botanical Heritage token rule — used only for inline form errors.

## Styling Approach

**Framework:** Tailwind CSS 3.4, utility-first, inline on every element. No `@apply`, no CSS modules, no styled-components.

**Design tokens (from `tailwind.config.ts`):**
- Colors: `parchment`, `cream`, `stone-white`, `forest` / `forest-dark`, `terracotta`, `warm-stone`, `dark-bark`, `warm-umber`, `sage-mist`
- Shadows: `shadow-warm` (resting), `shadow-warm-md` (hover/elevation)
- Fonts: `font-playfair` (headings, plant names), `font-inter` (default body, buttons, metadata)

**Do not use:** Generic Tailwind grays (`gray-*`, `slate-*`, `stone-*` except `stone-white`), cool shadows, or Tailwind blue/green defaults. Use the Botanical Heritage tokens exclusively.

**Shape language:**
- Cards and images: `rounded-2xl` (16px)
- Buttons and inputs: `rounded-lg` (8px)
- Badges and pills: `rounded-full`

**Responsive breakpoints:** `sm:`, `lg:` used for layout switches. The filter sidebar is `hidden lg:block`; mobile uses a collapsible drawer.

**Interactive states:**
- Hover on primary buttons: `hover:bg-forest-dark`
- Hover on secondary elements: `hover:text-dark-bark` or `hover:bg-stone-white`
- Destructive/remove hover: `hover:text-terracotta hover:bg-terracotta/10`
- Opacity-based disabling: `disabled:opacity-50 disabled:cursor-not-allowed`
- Transitions: `transition-colors duration-150` or `transition-colors duration-200`

**Typography scale examples:**
- Section label / eyebrow: `text-[11px] font-semibold uppercase tracking-[0.06em] text-warm-stone`
- Body text: `text-sm text-warm-umber`
- Plant card heading: `font-playfair text-xl font-semibold text-dark-bark`
- Page heading: `font-playfair text-3xl font-semibold text-dark-bark`

**Active tab class:** `text-forest border-b-2 border-forest` — use `toHaveClass(/text-forest/)` in tests.

## Import Organization

**Order (observed across files):**
1. React hooks — `import { useEffect, useState, ... } from 'react'`
2. Next.js utilities — `import { useRouter, useSearchParams } from 'next/navigation'`, `import Image from 'next/image'`, `import Link from 'next/link'`
3. Internal `@/` path-aliased imports — `import { createClient } from '@/lib/supabase/client'`
4. Type-only imports — `import type { Plant } from '@/lib/types'`
5. Relative imports (co-located components) — `import NavUser from './NavUser'`

No Prettier config; `next lint` enforces style. Imports are not grouped with blank lines in most files — they run consecutively.

**Path alias:** Always `@/` for cross-directory imports. Relative `./` only for co-located files in the same directory.

## Error Handling

**Server components:** Errors from Supabase are implicitly ignored on reads; missing data triggers `notFound()` or `redirect()`. No try/catch wrapping of Supabase calls in RSCs.

**Client components:** Supabase mutation errors captured from the returned `{ error }` and set to local state:
```typescript
const { error } = await supabase.from('plant_lists').insert(...)
if (error) { setError(error.message); return }
```

**Auth callback route (`app/api/auth/callback/route.ts`):** Handles both `token_hash` (preferred) and `code` (PKCE fallback). On any error, redirects to `/auth/login?error=auth`. No thrown exceptions.

**Empty catch:** `lib/supabase/server.ts` has an intentional empty `catch {}` in `setAll` to suppress cookie-setting errors from RSC render (expected behavior per Supabase SSR docs).

**No global error boundaries:** No `error.tsx` files found in the route tree.
