## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| migration SQL → live production Postgres | `supabase db push` applies DDL + seed to the live catalog by reference |
| anonymous client → plant_relationships RLS | Public SELECT only; no public write path |
| seed name-resolution → plants table | Untrusted name strings resolved via ILIKE; ambiguity/absence must fail loud, not silently |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-03-04 | Tampering / Elevation | `plant_relationships` world-writable if RLS omitted | mitigate | Explicit `ALTER TABLE plant_relationships ENABLE ROW LEVEL SECURITY` + SELECT-only `USING (true)` policy; NO INSERT/UPDATE/DELETE policy — writes are service-role/migration only (mirrors `plants`). Acceptance criterion greps that no `FOR INSERT/UPDATE/DELETE` policy exists. |

| T-03-06 | Repudiation / false-pass | Silent empty seed (bare `INSERT ... SELECT ... WHERE ILIKE` inserts 0 rows on missing plant) | mitigate | `pg_temp.resolve_plant` + `SELECT id INTO STRICT` RAISEs `NO_DATA_FOUND`/`TOO_MANY_ROWS`, naming the offending plant; whole `DO` block rolls back in the migration transaction (D-05). | T-03-07 | Integrity | SQL injection surface in the seed name resolution | mitigate | Resolution uses parameterized PL/pgSQL function args (`p_name`/`p_latin` bound into the query), not string-built SQL; names are static literals authored from the SEED-MANIFEST, not runtime user input. |
