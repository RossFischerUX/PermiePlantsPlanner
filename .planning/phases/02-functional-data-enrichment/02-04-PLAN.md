Close the Phase 02 verification blocker and three warnings so DATA-05 (BLOCKED) and DATA-02 (PARTIAL for succession_role) become genuinely satisfied.

The phase shipped (plans 02-01, 02-02, 02-03) but `--verify` reported a hollow green: it used `.not(f,'is',null)` for required array fields, so empty Postgres arrays (`{}`) counted as covered while carrying no data. At verification time (catalog = 1716 rows): 129 plants had `succession_role = '{}'`, 4 had `propagation_methods = '{}'`, 8 had `permaculture_uses = '{}'`. Root cause: `validArr` writes `[]` (not `null`) when Claude returns only out-of-vocab tags for required array fields, and `[]` is not NULL so the OR-of-nulls targeting never re-selects those rows — the empty state is permanently locked in with a green gate.

Note: the catalog was de-duplicated after verification (quick task 260518-vxy: 1716 → 1455 rows). The exact affected-row counts will differ at execution time; this plan proves closure via live DB queries, not hardcoded counts.

Purpose: make the DATA-05 gate honest and re-enrich the affected rows so succession data exists for the whole catalog.
Output: hardened `scripts/enrich-functional-data.ts`, an unconditional Playwright assertion, and DB-query proof that zero required-array fields are empty or null.
